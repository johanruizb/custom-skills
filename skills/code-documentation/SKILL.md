# Code Documentation Generator

## Overview

This skill generates and maintains useful inline documentation directly in source files. It does not add obvious or redundant comments — it documents purpose, behavior, dependencies, contracts, edge cases, and design decisions that cannot be inferred from reading the code alone.

It works in two modes (incremental update or full regeneration), follows language-native conventions (docstrings, JSDoc, TSDoc, PHPDoc, Rustdoc, JavaDoc), and validates that no functional behavior was changed.

## When to Use

- User asks to "documentar", "generate documentation", "add docstrings", "comment the code", "mejorar la documentación".
- User wants to update stale or incomplete inline docs across a project.
- User wants a full documentation regeneration from scratch.
- User needs documentation audits (find obsolete, incorrect, or redundant docs).

Don't use for:
- Generating external docs (README, wiki, API site) — use `markdown-document-enhancement` or similar.
- Single-file quick edits — just do it directly, no need for the full workflow.
- Generating docs for generated/vendored/third-party code.

## Pre-flight: Mode Selection

Before modifying anything, ask the user which mode to use via `clarify`:

**Question:** "¿Qué modo de documentación prefieres?"
**Choices:**
1. "Actualización incremental — conserva documentación válida, corrige la incompleta, elimina solo la que no aporta valor"
2. "Regeneración completa — elimina y reescribe toda la documentación (requiere confirmación antes de eliminar)"

If the user already specified the mode in their request, skip this step.

## Phase 1: Analysis

Goal: understand the project before touching anything.

1. **Scan project structure.** Use `search_files(target='files')` to list all source files. Identify directory layout, entry points, test directories, config files.

2. **Detect languages and frameworks.** Read manifest/config files:
   - Python: `pyproject.toml`, `setup.py`, `requirements.txt`, `Pipfile`
   - JS/TS: `package.json`, `tsconfig.json`, `deno.json`
   - PHP: `composer.json`
   - Rust: `Cargo.toml`
   - Java: `pom.xml`, `build.gradle`
   - Go: `go.mod`
   - Ruby: `Gemfile`
   - C#: `.csproj`, `.sln`

3. **Identify exclusion targets.** Mark these as DO NOT MODIFY:
   - `node_modules/`, `vendor/`, `dist/`, `build/`, `__pycache__/`, `.venv/`
   - Generated code (protobuf stubs, OpenAPI generated clients, migration files)
   - Lock files (`package-lock.json`, `poetry.lock`, `Cargo.lock`)
   - Third-party or vendored code
   - Files with headers indicating generation (e.g., `// Code generated by...`, `# DO NOT EDIT`)

4. **Assess existing documentation.** Read a sample of files per module. Classify existing docs as:
   - **Valid** — accurate, useful, matches code
   - **Stale** — references renamed/removed functions, wrong types, outdated behavior
   - **Redundant** — restates what code already says (e.g., `// increment i by 1` for `i += 1`)
   - **Superficial** — exists but adds no insight (e.g., `// helper function`)
   - **Missing** — complex logic with no documentation
   - **Critical** — documents design decisions, warnings, constraints — PRESERVE

5. **Detect available tools.** Check what validators exist:
   ```bash
   # Run these to detect what's available
   which ruff flake8 pylint mypy eslint tsc prettier rustc cargo go vet rubocop phpstan 2>/dev/null
   ```
   Record which ones are available for Phase 4 validation.

**Completion criterion:** You have a structured map of the project: languages, frameworks, file classification, doc quality assessment, available tools, and exclusion list.

## Phase 2: Proposal

Present the plan to the user before making changes. Use `clarify` or print a summary and wait for confirmation.

The proposal must include:

1. **Project summary** — what the project does, in 1-2 sentences
2. **Languages and frameworks detected**
3. **Recommended documentation style** per language (see `references/language-conventions.md`)
4. **Files to modify** — grouped by module/directory with count
5. **Files excluded** — with reason (generated, vendored, etc.)
6. **Mode selected** — incremental or full regeneration
7. **Risks** — files with complex logic where docs could be wrong, large files, files with existing critical comments
8. **Tool availability** — which linters/formatters/compilers will run in validation

Wait for explicit confirmation before proceeding. If the user says "procede" or similar, continue.

**Completion criterion:** User has confirmed the proposal and mode.

## Phase 3: Implementation

Process the project in controlled batches by module or directory. Never process the entire project in one pass.

### Batch strategy

- Group files by module/directory (e.g., `src/auth/`, `src/models/`, `src/api/`).
- Process one batch at a time.
- After each batch, run available validators (lint, type-check) to catch errors early.
- Use `delegate_task` for parallel processing of independent modules when the project is large (3+ independent modules).

### Documentation generation rules

For each file, generate documentation that prioritizes:

- **Purpose** of the file, module, class, function, or component
- **Responsibilities** — what this unit is accountable for
- **Parameters** — names, types, semantics (not just `str` but "user email, must be validated")
- **Return values** — type and meaning
- **Exceptions/errors** — what can be raised/thrown and when
- **Side effects** — I/O, mutations, state changes, event emissions
- **Dependencies** — critical external modules, services, APIs
- **Preconditions/postconditions** — what must be true before/after
- **Constraints** — rate limits, max sizes, supported formats
- **Edge cases** — empty input, null, boundary values, concurrent access
- **Design decisions** — non-obvious choices, trade-offs, historical context
- **Public contracts** — interfaces, extension points, API stability

AVOID:
- Restating what code already expresses (`// sets x to 5` for `x = 5`)
- Commenting every line
- Documenting trivial implementations (getters/setters, one-line wrappers)
- Inventing behaviors not verifiable from the code
- Changing code logic (only fix syntax errors you introduce)
- Generic boilerplate blocks repeated across files
- Documenting secrets, credentials, or sensitive values

### Incremental mode rules

- Keep valid documentation unchanged
- Fix stale docs to match current code
- Expand superficial docs to add real insight
- Remove only value-less comments (pure restatement, obvious labels)
- Add missing docs where complexity warrants
- NEVER remove comments that document decisions, warnings, constraints, or historical context

### Full regeneration mode rules

- Requires explicit user confirmation (already obtained in pre-flight)
- Remove replaceable comments (restatements, obvious labels, generic blocks)
- Regenerate all documentation from scratch using code analysis
- PRESERVE indispensable comments: design decisions, warnings, constraints, non-obvious behavior, historical context, security notes, TODO/FIXME with rationale
- When unsure whether a comment is indispensable, keep it

### Subagent delegation

For large projects (50+ source files), use `delegate_task` to parallelize:

- Each subagent gets: the module path, the documentation rules (from this skill), the language conventions (from `references/language-conventions.md`), the mode, and the exclusion list.
- Each subagent processes one module and returns a summary of changes.
- The main agent consolidates results and checks for consistency.

Example delegation context:
```
Module: src/api/
Language: TypeScript
Convention: TSDoc (see references/language-conventions.md#typescript)
Mode: incremental
Rules: [paste the Documentation generation rules section]
Exclusions: [paste exclusion list]
Available tools: eslint, tsc, prettier
```

**Completion criterion:** All batches processed, each batch validated with available tools, no new lint/type/compile errors introduced.

## Phase 4: Validation

After all modifications:

1. **Run formatters/linters:**
   ```bash
   # Python
   ruff check . && mypy src/ 2>/dev/null
   # JS/TS
   npx eslint . --quiet && npx tsc --noEmit 2>/dev/null
   # Rust
   cargo check 2>/dev/null
   # Go
   go vet ./... 2>/dev/null
   ```

2. **Verify no functional changes:**
   ```bash
   # If git is available
   git diff --stat
   # Check that only comments/docstrings changed, no logic
   git diff -U0 -- '*.py' | grep -E '^[+-]' | grep -vE '^[+-]\s*(#|"""|\*|//|/\*|\*/)' | grep -vE '^[+-]{3}'
   ```

3. **Run tests if available:**
   ```bash
   pytest 2>/dev/null || npm test 2>/dev/null || cargo test 2>/dev/null || go test ./... 2>/dev/null
   ```

4. **Spot-check doc accuracy:** Read 3-5 modified files and verify the new docs match the actual code behavior.

5. **Report unrunnable validations:** If a tool is not installed or a test suite doesn't exist, state it explicitly — do not skip silently.

**Completion criterion:** All available validators passed (or failures pre-existed), no functional changes detected, tests pass (or were already failing for unrelated reasons).

## Phase 5: Final Summary

Deliver a report with:

| Category | Details |
|---|---|
| Files analyzed | N total, grouped by type |
| Files modified | N, listed by module |
| Documentation created | N new docstrings/comments added |
| Documentation updated | N stale/superficial docs improved |
| Documentation removed | N value-less comments deleted (mode-dependent) |
| Critical comments preserved | N, with file references |
| Files omitted | Listed with reason (generated, vendored, etc.) |
| Validations executed | Which tools ran, results |
| Validations not available | Which tools were missing |
| Errors/warnings | Any issues introduced or found |
| Recommendations | Follow-up suggestions (e.g., "module X has no tests", "Y has complex logic deserving a design doc") |

If git is available, show `git diff --stat` for a quick overview.

## Common Pitfalls

1. **Documenting trivial code.** A one-line getter does not need a docstring. Skip it. Documentation should add insight, not volume.

2. **Removing critical comments in full-regen mode.** Comments about race conditions, security constraints, "we tried X and it failed because Y" are indispensable. When in doubt, keep the comment.

3. **Inventing behavior.** If you cannot verify a behavior from the code, do not document it. State "behavior not verifiable from code" rather than guessing.

4. **Changing code logic.** Documentation changes only. If a syntax error is introduced by a misplaced comment, fix the comment placement — do not refactor the code.

5. **Ignoring existing conventions.** If the project uses Google-style docstrings, do not switch to Sphinx. Match what's already there. Check `references/language-conventions.md` for the standard formats, but defer to project style.

6. **Processing generated code.** Protobuf stubs, OpenAPI clients, migration files, and build outputs should never be modified. Always check for generation headers.

7. **Skipping validation.** Even if you think comments are harmless, run the linters. A malformed docstring can break documentation generators (Sphinx, TypeDoc, rustdoc).

8. **Not batching.** Processing 200 files in one pass makes errors hard to isolate. Batch by module, validate after each batch.

9. **Documenting secrets.** Never add comments that reference API keys, passwords, tokens, or connection strings, even if they appear in the code.

10. **Silent validation gaps.** If `mypy` is not installed, say so in the summary. The user needs to know what was NOT checked.
