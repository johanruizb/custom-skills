# Test Smells Checklist

Systematic checklist for evaluating every test in a suite. Use during Phase 4. For each test file, walk through every category and record findings.

## How to Use This Checklist

For each test file:
1. Read the test file completely.
2. Read the source file(s) it tests (to verify the test matches current behavior).
3. Walk through every category below.
4. Classify each test as: **keep** (valuable), **fix** (valuable but broken/fragile), **remove** (no value), **split** (valuable but tests too many things).
5. Record: file path, test name, classification, category that triggered it, evidence (specific code snippet or line number).

---

## 1. No Useful Assertions

**Symptom**: The test calls production code but doesn't verify the outcome, or verifies only that no exception was thrown.

**Patterns to search for**:
- `assert True`, `assert result is not None` (without further checks)
- `try: ... except: pass` — swallows exceptions without asserting
- Tests that end without any assert statement
- `expect(result).toBeDefined()` without further checks
- `assert_no_response(resp)` — checks only that no error was raised
- Tests with only `pass` as the body

**Search patterns**:
- Python: `def test_.*\n.*\n.*pass$` — empty test bodies
- JavaScript: `it\(.*\)\s*=>\s*{[^}]*}` — no expect() inside
- Python: `assert result is not None$` — weak assertion as the only check

**Decision**: Remove if the test verifies nothing meaningful. Fix if the test has the right setup but is missing assertions — add the assertions it should have.

---

## 2. Implementation-Detail Testing

**Symptom**: The test verifies internal method calls, private state, or structural details that would break on a valid refactor.

**Patterns**:
- Asserting that a specific private method was called: `assert mock._internal_method.called`
- Asserting the number of times a method was called when the count is an implementation detail
- Asserting on the order of internal operations
- Testing the return value of a private method
- Asserting on SQL queries generated by an ORM
- Asserting on the exact structure of an internal data object

**Decision**: Remove or rewrite to test observable behavior. If a method is private, test it through the public interface. If you need to verify an interaction, assert that the *effect* happened (data saved, message sent), not that a specific method was called.

**Exception**: Testing that a specific side effect occurs (e.g., "an email was sent") is valid if the side effect is part of the contract.

---

## 3. Duplicate / Redundant Tests

**Symptom**: Two tests verify the same behavior with the same inputs and the same assertions.

**Patterns**:
- Same test name with slight variations (`test_user_creation` and `test_create_user`)
- Tests with different names but identical setup, execution, and assertions
- Tests that test the same code path but with trivially different fixture data that doesn't change the behavior
- A test that is a subset of another test's assertions

**Decision**: Keep the most descriptive one. Remove the duplicate. If two tests test overlapping but different aspects, keep both but clarify the names.

**Detection**: Group tests by the function/behavior they test. Within each group, compare setup + assertions. If identical → duplicate.

---

## 4. Fragile Tests

**Symptom**: The test passes or fails depending on execution order, shared state, timing, or external conditions.

**Patterns**:
- Tests that modify a shared database without cleaning up
- Tests that depend on `datetime.now()` without freezing the clock
- Tests that depend on the order of records in a query without sorting
- Tests that rely on environment variables being set
- Tests that depend on external services (real API calls, real SMTP, real Redis) without mocking
- Tests that depend on the filesystem state (existence of temp files, CWD)
- Tests that fail intermittently ("it passed yesterday") — classic flaky test
- Tests using `time.sleep()` to wait for async operations — race condition

**Search patterns**:
- `datetime.now()` / `time.time()` without a mock/freezer
- `time.sleep(` — often a sign of a race condition
- Global/module-level mutable state modified in tests
- `os.environ` read without being set in the test
- Real HTTP calls (`requests.get('http://`) without mocking

**Decision**: Fix. Isolate the test: use a fresh database/transaction, freeze time, mock external services, sort query results, use deterministic fixtures. If the test cannot be made deterministic, remove it and document the gap.

---

## 5. Excessive or Incorrect Mocking

**Symptom**: The test mocks so much that it validates only the mock, not the real code.

**Patterns**:
- Mocking the system under test itself (mocking the function being tested)
- Mocking everything — every dependency, every collaborator — so the test is pure mock-to-mock interaction
- Mocks that return mocks that return mocks (`mock.return_value.mock.return_value`)
- Mocking types/interfaces that don't match the real implementation (mock will pass but real code would fail)
- `MagicMock()` used as a database model — the mock will never raise IntegrityError
- Mocking a dependency that is trivially testable for real (e.g., mocking a pure function)
- Over-mocking: mocking a logger, mocking a config reader, mocking everything that isn't the SUT

**Decision**: Reduce mocking. Test the real behavior. Mock only external boundaries (APIs, databases, filesystem, email). If a dependency is cheap to use for real, use it for real. If the test only validates mock interactions, rewrite it to test real behavior or remove it.

---

## 6. Tautological Tests (Always Pass)

**Symptom**: The test passes regardless of whether the production code works.

**Patterns**:
- `assert True`, `assert 1 == 1`
- `assert result is not None` when the function always returns something
- `except Exception: assert True` — catches everything, always passes
- Tests that assert on a mock's return value that was explicitly set in the test: `mock.return_value = 42; assert func() == 42`
- Tests where the assertion is tautologically true given the setup: `x = 5; assert x == 5`
- Tests that catch an exception and assert it was raised, but the exception is always raised

**Decision**: Remove. These tests provide zero value and false confidence. If the behavior is important, write a real test that would fail if the code broke.

---

## 7. Obsolete Tests

**Symptom**: The test references code that no longer exists or behaves differently.

**Patterns**:
- Import of a function/class/module that was renamed or removed (the test would fail to import)
- Tests for a function whose signature changed (the test calls it with old arguments)
- Tests for a feature that was removed from the product
- Tests that assert on old default values, old API responses, old data formats
- Tests guarded by `@pytest.mark.skip` or `it.skip()` or `xtest` that were skipped long ago

**Detection**:
- Try to run the test — if it errors on import or signature mismatch, it's obsolete
- Check git history: `git log --oneline -- <test_file>` — when was it last updated?
- Check if the tested function still exists: search for its name in source
- Check if the tested behavior matches current source: read the implementation

**Decision**: Remove. If the behavior is still relevant but the test is outdated, rewrite it to match current behavior. Do NOT just update expected values without understanding why they changed.

---

## 8. Slow Tests Without Value

**Symptom**: Tests that take >5 seconds but don't test anything that requires that time.

**Patterns**:
- `time.sleep(5)` to wait for an async operation instead of using a proper wait/poll
- Tests that create a real HTTP server, make real network calls, and tear it down — when a mock or in-memory adapter would suffice
- Tests that run a full application bootstrap for a pure function test
- Tests that load large fixtures when small ones would test the same behavior
- Tests that do unnecessary I/O (file reads, DB queries) for logic that doesn't need it
- Tests in the unit test directory that are actually integration tests

**Detection**: Run the suite with timing output (`pytest --durations=10`, `jest --verbose`, `go test -v`). Look at the slowest tests. Evaluate whether the slowness is inherent or accidental.

**Decision**: Fix. Replace `time.sleep` with proper async waiting. Replace real network calls with mocks or in-memory adapters. Move true integration tests to a separate directory/suite. Use smaller fixtures. Don't remove a slow test if it's testing something genuinely important — but make it faster.

---

## 9. Missing Coverage

**Symptom**: Important functionality has no tests.

**What to look for** (high-priority gaps):
- Authentication and authorization logic (login, permissions, role checks)
- Input validation (form validation, API request validation, schema validation)
- Error handling paths (what happens when the DB is down, when the API returns 500, when the file is corrupted)
- Boundary conditions (empty list, single item, max items, negative values, null/None)
- Business rules (pricing, discounts, quotas, limits, calculations)
- External integrations (API calls, webhook handling, message queue consumers)
- Database operations (CRUD, transactions, migrations, constraints)
- Concurrency / async behavior (race conditions, timeouts, retries)
- API endpoints (status codes, response format, error messages)
- File processing (parsing, validation, transformation)
- Scheduled jobs / background tasks (what happens when they run, when they fail)
- Security-sensitive operations (password hashing, token generation, session management)

**Decision**: Add tests. Prioritize by risk: critical business rules > error paths > edge cases > happy paths. A missing test for a critical business rule is more important than a missing test for a utility function.

---

## 10. Other Smells

### Test Names That Don't Describe Behavior
- `test_1`, `test_stuff`, `test_it_works`, `test_helper`
- **Decision**: Rename to describe the behavior: `test_rejects_invalid_email`, `test_returns_404_for_unknown_user`.

### Tests That Test Too Many Things
- A single test with 50+ lines, multiple setups, multiple assertions on different behaviors
- "and" in the test name: `test_creates_user_and_sends_email_and_logs_event`
- **Decision**: Split into multiple focused tests.

### Tests That Depend on Other Tests
- A test that sets up state for the next test, with tests that fail if the setup test doesn't run first
- **Decision**: Fix. Each test must be independent. Use setUp/fixture to establish state.

### Tests That Test the Test Framework
- `assert pytest works`, `assert the assertion library is functional`
- **Decision**: Remove. These are meta-tests that add no value.

### Tests That Test Configuration
- Asserting that a config file has a specific key (unless the config is the product)
- **Decision**: Remove unless the config IS the behavior being tested.

## Classification Decision Tree

```
For each test:
1. Does it have meaningful assertions?
   NO → REMOVE (category 1 or 6)
2. Does it test observable behavior (not implementation details)?
   NO → REMOVE or REWRITE (category 2)
3. Is it a duplicate of another test?
   YES → REMOVE the weaker copy (category 3)
4. Is it deterministic and isolated?
   NO → FIX (category 4)
5. Does it mock the system under test or mock everything?
   YES → REWRITE or REMOVE (category 5)
6. Would it pass even if the code were broken?
   YES → REMOVE (category 6)
7. Does it reference code that still exists and behaves the same?
   NO → REMOVE or REWRITE (category 7)
8. Is it unnecessarily slow?
   YES → FIX (category 8)
9. Does it test one behavior with a descriptive name?
   NO → SPLIT or RENAME
10. Otherwise → KEEP
```

Record the final classification for each test:
```
file: tests/test_foo.py
test: test_bar
classification: remove
category: 6 (tautological)
evidence: line 23: assert True  # no assertion on actual behavior
```
